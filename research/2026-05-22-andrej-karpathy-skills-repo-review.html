What it is

The repo packages Andrej Karpathy-inspired observations about common LLM coding failures into reusable instruction files. The README frames the problem as agents making wrong assumptions, hiding confusion, overcomplicating code, touching unrelated code, and failing to define verifiable success criteria.

“The models make wrong assumptions on your behalf and just run along with them without checking. They don't manage their confusion, don't seek clarifications, don't surface inconsistencies, don't present tradeoffs, don't push back when they should.”Quote excerpted in repo README from Andrej Karpathy’s post

It contains:

Files that matter 14 files total
  • CLAUDE.md — the core guideline file.
  • skills/karpathy-guidelines/SKILL.md — directly portable to a Hermes-style skill.
  • .cursor/rules/karpathy-guidelines.mdc — Cursor always-apply project rule.
  • .claude-plugin/* — Claude Code plugin metadata.
  • EXAMPLES.md — concrete examples of bad vs better agent behavior.

How it can help us

The useful part is not the repo complexity — it is intentionally simple. The useful part is that it encodes exactly the coding-agent behavior we often want from subagents and coding profiles.

1. Think before coding
Surface assumptions

Good for repo work where an agent might otherwise silently pick the wrong interpretation and edit too much.

2. Simplicity first
Avoid abstraction bloat

Directly fights the “1000 lines when 100 would do” failure mode.

3. Surgical changes
Cleaner diffs

Useful for PRs and patches: every changed line should trace to the requested task.

4. Goal-driven execution
Tests + verification

Transforms vague coding tasks into success criteria and loops until checks pass.

5. Review lens
Better code review prompts

Can be used as a checklist for reviewing agent-generated diffs before merge.

6. Cross-agent consistency
Claude/Cursor/Hermes

Same behavioral rules can be shared across Claude Code, Cursor, and Hermes skills.

How I would use it for you

I would not install it as a blanket “always on for every task” instruction. Your Hermes already has tool-discipline, artifact-first reporting, and Telegram-specific communication rules. Adding this globally could cause too many clarification questions for obvious defaults.

→
Best setup: install/adapt it as a scoped Hermes skill named something like karpathy-coding-guardrails, then load it automatically for coding, code review, refactor, PR, and subagent-development tasks.
Recommended local integration safe path
  • Copy or adapt skills/karpathy-guidelines/SKILL.md into ~/.hermes/skills/software-development/.
  • Keep its trigger narrow: “writing, reviewing, debugging, or refactoring code.”
  • Do not let it override the artifact policy or Telegram brevity rules.
  • For coding subagents, mention this skill in the delegation context or attach it where supported.
  • For project repos, optionally merge its CLAUDE.md content into project-level AGENTS.md/CLAUDE.md only when that repo benefits from cautious diffs.

Caveats

What to be careful about important
  • It biases toward caution. The repo says this explicitly. Great for non-trivial code, annoying for tiny obvious fixes.
  • Potential conflict with “act, don’t ask.” Hermes should ask only when ambiguity genuinely changes the action; the skill’s “ask if uncertain” should be scoped to real uncertainty.
  • No dedicated LICENSE file found. README and SKILL frontmatter say MIT, but the repo did not expose a root LICENSE file via GitHub contents.
  • It is mostly instructions, not tooling. The benefit comes from better agent behavior, not new functionality.

Verdict

Yes — helpful, but narrowly. This is worth adapting into our Hermes coding workflow because it reinforces exactly the behaviors we want from coding agents: minimal diffs, clear assumptions, no speculative abstractions, and verified success criteria.

My recommendation: install/adapt it as a Hermes coding skill, not as a global personality. Use it for code changes, PR review, debugging, refactoring, and delegated coding work.
