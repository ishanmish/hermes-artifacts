Implemented direction

Hermes outputs should become interfaces, not walls of chat.

The Claude Code HTML article validates the direction: Markdown is useful as a substrate, but HTML is the better human review surface. I implemented the first reusable publisher so future big outputs can become indexed, clickable artifacts instead of long Telegram messages.

What changed now

1 · publisher

Reusable artifact publisher

Created artifact_publisher.py. It takes an HTML file, category, summary, tags, and status, then publishes it to the stable GitHub Pages artifact hub.

2 · manifest

Artifacts are indexed

The hub now has an artifacts.json manifest with title, URL, category, tags, created time, status, expiry, and summary. This solves future findability.

3 · hub

Clickable artifact hub

The index page is regenerated as a dark dashboard with cards, categories, and manifest link. Future reports will show up there automatically.

The new output contract

1. TaskResearch, report, comparison, code review, trading idea, travel plan.
2. StructureExtract verdict, sections, source trail, metadata, next action.
3. ArtifactGenerate a readable HTML interface, not a prettier transcript.
4. PublishPush to GitHub Pages and update the artifact manifest/index.
5. TelegramSend only the clickable URL, short summary, and recommendation.

When to use what

Chat

Use for: short answers, status, links, quick decisions.

Shape: 3–8 bullets, no tables, no walls.

HTML

Use for: big reports, comparisons, dashboards, research, code reviews, travel plans, backtests.

Shape: cards, sections, source trails, charts/tables, actions.

Data

Use for: reuse by agents and cleanup.

Shape: JSON/Markdown under the hood; HTML remains the reading layer.

Artifact types

Recurring

Stable dashboards

Same URL updates each run. Example: Reddit digest. Optional recent history on-page.

One-off important

Saved artifacts

Permanent page in the artifact hub. Good for travel research, strategy memos, code reviews, major comparisons.

Temporary

Auto-expiring artifacts

Published with an expiry date in the manifest. Cleanup command removes expired files later to avoid bloat.

Design rules for future artifacts

Decision-oriented

Every artifact should answer: what decision does this help Ishan make? It needs a verdict, confidence, tradeoffs, and next action.

Readable once

Optimize for someone opening it once and understanding the point quickly. Use hierarchy, cards, labels, and source links.

Self-contained

Inline CSS, responsive layout, stable URL, enough context to make sense months later.

Actionable

Where useful, add copy/export buttons, filters, collapsible sections, or comparison controls. Interactivity must reduce thinking.
